# 把创意留给 LLM，把结构交给代码：模板创建技能的诞生记

> Day 41 · 2026-03-12

昨天还在纠结怎么让新用户跨过 frontmatter、区段结构和变量表这座大山，今天决定不再堆砌文档，而是直接造一个“带路党”。我们要开发的 `prism-template-author` 技能，核心思路很简单：让 LLM 去搞定充满创意的 Prompt 内容，而把枯燥的结构决策固化成流程。

## 为什么我不打算只写个 CLI 工具

一开始我也想过，搞个命令行工具，用户输几个参数，噼里啪啦生成一堆文件不就完了？但转念一想，模板创建最难的不是“生成文件”，而是“决策”。

用户得想清楚：这篇产出是写给谁看的（叙事复盘还是结构化教程）？是一个 KL 一篇还是整个视角一篇？需不需要分阶段生成？这些认知负荷如果全压在用户头上，门槛太高；如果全扔给 LLM 自由发挥，又容易跑偏。

所以我决定采用“决策引导 + Schema 参考 + Scaffold 工具”的三层架构。主文件 `SKILL.md` 里只放四个关键问题（产出形态、写作风格、生成复杂度、质量审校），通过这四个问题直接把用户意图映射到具体的 `split`、`type` 和组件配置上。至于详细的 frontmatter 字段表、变量表这些几百行的干货，全部塞进 `schema-reference.md` 附件里。

这就好比带徒弟：师傅（SKILL.md）只问关键方向，具体的招式手册（Schema）放在手边随时查，绝不在第一天就把整本百科全书拍在徒弟脸上。这种渐进式披露策略，让主技能文件死死控制在 130 行以内，交互界面清爽多了。

## 让 Scaffold 工具做个“莫得感情”的建筑工

架构定好后，最关键的执行环节来了。我定义了两个 AI 工具：`prism_scaffold_template` 和 `prism_scaffold_component`。

这里我做了个强硬的决定：这两个工具必须是纯确定性的，严禁调用 LLM 生成任何内容。它们的职责只有一个——校验命名、合并默认值、生成 frontmatter 和区段占位符。比如运行 `prism_scaffold_template({ name: "tutorial", split: "per-perspective", type: "blog" ... })` 时，它只管把架子搭得严丝合缝，连一个字的正文都不写。

刚开始实施时，我还担心这样会不会太死板？但跑了几次 `--dry-run` 验证后发现，这正是我们需要的。结构正确性由代码保证，绝对不出错；而内容填充的创意工作，完全交给 Agent 根据前面的决策引导去发挥。这种“建筑工”与“设计师”的职责剥离，反而让生成的模板既规范又有灵魂。

另外，文件写入路径也做了智能处理：优先往知识库的 `outputs/_templates/` 目录写，如果没有这个目录，再回退到项目内置目录。这种本地优先的策略，让生成的模板能直接融入用户的工作流，而不是孤零零地躺在系统文件夹里吃灰。

## 今天的收获

- 高门槛任务不应仅提供 CLI 工具，而应设计专用技能通过“四问映射”引导用户完成结构化决策。
- 技能架构需将决策引导逻辑与详细 Schema 参考分离，利用渐进式披露避免信息过载。
- Scaffold 工具必须严守纯确定性逻辑，仅负责结构校验与骨架生成，严禁越俎代庖生成内容。
- 创意任务（Prompt 编写）应留给 LLM，而配置决策应通过固化流程由技能引导完成。
- 建立“本地知识库优先、项目目录回退”的文件写入机制，能更自然地融入用户工作流。
